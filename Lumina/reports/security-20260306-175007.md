## High-Risk Native Surfaces

1. `lumina_evolve_app`
   - Directly edits files based on a provided replacement pattern.
   - Risk: arbitrary self-modification without authorization boundaries.

2. `shell_execute`
   - Opens arbitrary commands or URLs through the OS shell.
   - Risk: unexpected command execution or phishing surfaces if untrusted input reaches this path.

3. `get_recent_files` / `search_files`
   - Enumerate user files from common directories.
   - Risk: privacy expectations must be explicit and tested.

4. `capture_vision_syndrome`
   - Captures display content.
   - Risk: sensitive on-screen data exposure.

5. `lumina_web_search` and updater endpoint
   - Create non-local network egress paths.
   - Risk: inconsistent with offline-first claims.

## Input/Sanitization Notes

- `scrub_pii(...)` is now isolated in `src-tauri/crates/lumina-privacy` with dedicated unit coverage.
- Vault persistence is now isolated in `src-tauri/crates/lumina-vault` with tests proving PII redaction happens before storage.
- Query/search/file/shell surfaces still need broader policy tests that prove allowed input and blocked input behavior.

## Operational Notes

- Windows packaging now succeeds, but only after exposing a root-level `src-tauri` alias to the nested app tree.
- That alias is a build-system workaround, not a security control, and should be replaced with a cleaner repo layout or upstream-safe packaging fix when feasible.

## Required Follow-Ups

1. Add policy tests for sensitive commands.
2. Gate sensitive commands behind explicit UI consent where missing.
3. Define policy for network-enabled features versus air-gapped mode.
4. Continue decomposing the backend so more of the native surface is testable without the full Tauri shell.
